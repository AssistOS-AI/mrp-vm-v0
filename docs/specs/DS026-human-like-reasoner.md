# DS026 HumanLikeReasoner

## Introduction

`HumanLikeReasoner` is the first implemented solver-style external interpreter in MRP-VM v0. It is designed for bounded human-scale reasoning tasks where the problem can be represented explicitly enough to justify local symbolic work, but where the decomposition still benefits from an LLM-generated orchestration program.

The interpreter is external rather than native because it behaves like a distinct execution surface with its own contract, guidance assets, and routing decision. It accepts either a direct reasoning task or a structured brief produced by `logic-eval`, asks the managed LLM path to generate a bounded JavaScript reasoning program, validates that program, runs it inside a restricted runtime, and emits the resulting answer back into the ordinary family system.

## Core Content

### Input contract

`HumanLikeReasoner` must accept these repository-owned input shapes:

1. a direct natural-language bounded reasoning task,
2. a structured rewrite brief whose main payload is `rewritten_problem`,
3. a JSON envelope that may override `result_mode`, `program`, or generation settings for tests and controlled execution.

The interpreter must preserve explicit constraints and requested answer sections when they are already present in the input. It must not broaden the task into a generic essay request.

### Program-generation and execution flow

The execution flow is:

1. normalize the input into one bounded reasoning task,
2. obtain prompt guidance from default KUs,
3. ask the managed adapter for a bounded JavaScript reasoning program through `logicGeneratorLLM`,
4. run preflight validation on the returned source,
5. execute only approved constructs inside a restricted runtime,
6. emit one final answer through `ctx.emit(...)`,
7. return the emitted answer as the interpreter outcome.

Inline `program` input remains valid for tests and controlled fixtures, but the repository default path is generated program source plus local validation.

### Runtime contract

The generated reasoning program must create an `ExecutionContext` and may use only approved classes plus small scalar utilities. The repository-owned baseline allows:

| Surface | Role |
| --- | --- |
| `ExecutionContext` | Working memory, result storage, trace, assumptions, and final emission |
| `TextBuilder` through `ctx.text(...)` | Controlled answer construction |
| `RuleProblem` | Finite rule closure |
| `ConstraintProblem` | Finite assignment and exclusion |
| `GraphProblem` | Paths, reachability, and topological order |
| `SearchProblem` | Small bounded state-space planning |
| `NumericProblem` | Bounded integer reasoning |

The repository-owned prompt assets must teach the canonical public method names for these classes. The runtime may tolerate a small compatibility layer for common synonymous method names generated by models, but the canonical contract remains the documented class surface rather than invented helper objects.

`HumanLikeReasoner` shares the broad local-context discipline used by `AdvancedReasoner`: bounded local state, explicit trace, and inspectable stored results. The important difference is the final exit surface. `HumanLikeReasoner` ends by emitting a family value through `ctx.emit(...)`, while `AdvancedReasoner` ends by returning a typed bounded response through `ctx.returnResponse(...)`.

The program must not use `import`, `export`, `require`, `process`, `globalThis`, `eval`, `Function`, async constructs, or unbounded loops. Bounded reasoning in v0 is implemented through approved classes and explicit context operations, not through arbitrary general-purpose scripting.

### Output contract

The interpreter may emit:

1. plain text, or
2. structured output whose root still contains the final emitted text together with trace-supporting metadata.

Every successful output must record interpreter origin metadata so trace and evaluation can distinguish `HumanLikeReasoner` work from ordinary wrapper prose.

### Shipped reasoning-class baseline

The repository-owned implementation baseline in v0 covers these classes only:

1. `RuleProblem`
2. `ConstraintProblem`
3. `GraphProblem`
4. `SearchProblem`
5. `NumericProblem`

The broader research direction may later add more reasoning classes, but DS026 must describe only the shipped baseline as implemented today.

## Decisions & Questions

Question #1: Why is `HumanLikeReasoner` an external interpreter instead of another native command?

Response: It has its own routing identity, guidance assets, prompt discipline, and execution boundary. Treating it as an external interpreter keeps those concerns distinct from the native-command layer and aligns it with DS020.

Question #2: Why use a bounded JavaScript reasoning program instead of a single monolithic solver formalism?

Response: Human-scale reasoning tasks often mix several local forms such as assignment, graph reachability, small search, and bounded numeric checks. A bounded orchestration program lets the interpreter combine those classes explicitly without pretending that one solver formalism should absorb every case.

Question #3: Why does DS026 limit itself to five shipped reasoning classes instead of copying the full research wish list into the implementation contract?

Response: The DS set must remain defensible against the current repository, not against a future research direction alone. Listing only the shipped classes keeps the contract honest while still leaving room for later extension.

Question #4: Why allow a compatibility layer for synonymous method names if the prompt assets already define the canonical API?

Response: Real provider generations sometimes drift toward predictable names such as `addFact` or `addNode` even when the prompt asset teaches the preferred form. Tolerating a narrow set of synonyms makes the interpreter more robust, but the DS still treats the documented canonical methods as the primary contract so prompt assets and examples stay stable.

Question #5: Why keep `HumanLikeReasoner` on `ctx.emit(...)` instead of changing it to the `ReasonerResponse` exit style used by `AdvancedReasoner`?

Response: `HumanLikeReasoner` is the exact bounded-solver surface that primarily emits one solved answer value, optionally with structured metadata around that value. `AdvancedReasoner` exists specifically to expose richer bounded statuses such as `needs_engine`, `needs_review`, and `needs_clarification`. Preserving the different exit surfaces keeps the two contracts honest instead of flattening them into one ambiguous response model.

## Conclusion

`HumanLikeReasoner` is the implemented bounded reasoning interpreter for MRP-VM v0. It turns direct tasks or rewrite briefs into validated reasoning programs, executes them under strict runtime rules, and emits visible reasoning outcomes without duplicating the planner or the native-command layer.
