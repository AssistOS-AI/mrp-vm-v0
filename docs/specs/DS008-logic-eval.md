# DS008 logic-eval

## Introduction

`logic-eval` is the bounded symbolic solver runtime of MRP-VM v0. Its purpose is to separate natural-language understanding from verifiable symbolic reasoning. An LLM translates a user request into a narrow internal program called `SolverProgram`. The runtime then validates that program, executes it inside an explicit `ExecutionContext`, delegates local subproblems to specialized leaf solvers, and formats the structured result back into text. The public API remains minimal: one function receives an LLM adapter and a text, and returns a text. Internally, the same request passes through four strict phases: program construction, preflight validation, symbolic execution, and final formatting.

At the native-command boundary, `logic-eval` may receive either a bounded natural-language problem statement or an explicit JSON envelope that carries `program_steps`. Legacy `use` / `when` / `then` rule blocks may still be accepted as compatibility shorthand for small local guards, but the normative DS surface is the symbolic solver runtime described below.

The design is intentionally modest. It does not attempt to be a universal solver. It targets finite or moderate problems that a human formalizes, simplifies, and solves locally through explicit steps. Everything after the initial LLM translation must be direct, inspectable, and implementable: structural validation, early error detection, missing-information detection, contradiction detection, complexity estimation, and execution under strict time and step budgets. Controlled, honest failure is part of the design, not an exception.

The architecture rests on three internal pillars:

1. `SolverProgram` — the internal DSL program generated by the LLM that orchestrates steps.
2. `ExecutionContext` — typed working memory for the current request.
3. Leaf solvers — specialized local engines for rules, constraints, graphs, search, and bounded numeric problems.

Above the core sits `PreflightAnalyzer`, which decides whether execution is worth starting. Below the core sits a formatter LLM, which explains only what the runtime produced.

This DS records the full contract of the symbolic solver layer, including the DSL grammar, the context register schema, the leaf solver interfaces, the preflight verdict model, the fail-fast budgets, the end-to-end execution flow, and the prompt boundaries for the LLM stages.

## Core Content

### 1. Public interface and execution statuses

The external interface is a single conceptual function:

```
solveText(llmAdapter, text) -> text
```

Internally, every execution produces a standardized result object with a clear status, final data, and optional diagnostics. Without this standardization, the formatter cannot work robustly.

The standard statuses are:

| Status | Meaning |
| --- | --- |
| `solved` | The problem was formalized and solved within budget. |
| `needs_clarification` | Structural information is missing and questions were generated. |
| `invalid_program` | The generated program violates the DSL or approved references. |
| `unsat_early` | A contradiction was detected before serious search. |
| `too_complex` | Symbolic analysis or a solver decided the budget would be exceeded. |
| `failed` | The problem could not be formalized or executed within current limits. |
| `partial` | Only a partial solution or a useful reduction was obtained, not a complete verdict. |

Every leaf solver must return an object containing at minimum `status`, `solution` (or the local equivalent such as `facts`, `path`, or `plan`), `stats`, and `diagnostics`. Every final result must be honestly expressible as text, including when the conclusion is that the problem cannot be solved within budget.

### 2. ExecutionContext registers and typing

`ExecutionContext` is mandatory. Without it, the system would be reduced to isolated solver calls. With it, the internal program can keep variables, lists, intermediate results, admitted hypotheses, errors, and execution traces. This working memory must be logically typed, not merely textual. An explicit user-given fact is not treated the same as a derived value or a temporary assumption.

The minimum internal register schema is:

| Register | Content |
| --- | --- |
| `input` | The original problem and output instructions. |
| `given` | Explicit information extracted from the request. |
| `vars` | Intermediate variables and collections. |
| `results` | Results produced by solvers. |
| `assumptions` | Hypotheses explicitly introduced by the program, if the mode allows them. |
| `unknown` | Obligations or information recognized as missing. |
| `conflicts` | Contradictions detected before or during execution. |
| `trace` | Steps relevant to the final explanation. |
| `final` | The final object that will be formatted into text. |

The context API must be small but complete. It is an operational workspace, not just a variable dictionary.

| Method | Role |
| --- | --- |
| `set(name, valueOrExpr)` | Writes a variable into context. |
| `get(refOrExpr)` | Reads a value or evaluates a simple expression. |
| `has(name)` | Checks whether a variable exists. |
| `append(name, valueOrExpr)` | Adds an element to a list. |
| `extend(name, listExpr)` | Concatenates one list onto another. |
| `project(name, fromExpr, selectorExpr)` | Extracts a projection from a structure. |
| `filter(name, fromExpr, predicateExpr)` | Filters a collection. |
| `map(name, fromExpr, mapperExpr)` | Transforms a collection. |
| `unique(name, fromExpr)` | Removes duplicates. |
| `sort(name, fromExpr, keyExpr)` | Sorts a collection. |
| `count(name, fromExpr)` | Stores the cardinality of a collection. |
| `assert(expr, message)` | Checks a condition and halts execution if false. |
| `assume(label, valueOrExpr)` | Saves an explicit auxiliary hypothesis. |
| `storeResult(name, result)` | Saves a solver result. |
| `result(name)` | Reads a previously stored solver result. |
| `setFinal(valueOrExpr)` | Declares the final value to be formatted into text. |

Expressions inside these methods must remain simple and finite. An expression may be a path reference such as `$name`, `$name.field`, or `$name[index]`, or a small tree of access, comparison, and collection operators. The expression language is not general JavaScript; it is a bounded access and transformation language.

### 3. Internal DSL for SolverProgram

The internal DSL is JavaScript only at the syntactic surface. Semantically it is a narrow orchestration language for reasoning. The program may construct subproblems, read and write the context, project results, perform simple branching, and choose the final answer. It may not introduce new algorithms, uncontrolled loops, or arbitrary runtime access.

The required construction families are:

| Family | Description |
| --- | --- |
| Solver construction | Instantiation of `RuleProblem`, `ConstraintProblem`, `GraphProblem`, `SearchProblem`, or `NumericProblem`. |
| Context operations | `set`, `get`, `project`, `filter`, `map`, `assert`, `storeResult`, `setFinal`, plus `append`, `extend`, `unique`, `sort`, `count`, `has`, `assume`. |
| Small procedural control | `if`, `return`, and bounded iteration over finite lists. |
| Expressions | References, comparisons, counts, projections, filters, and simple maps. |
| Resolution | Calls to `solve()` on local subproblems. |
| Response construction | Selection and composition of the final value passed to the formatter. |

`SolverProgram` is an internal class with a single semantic entry point, `run(ctx)`. The program is generated by the LLM, but it defines no new primitives. It uses only the runtime-approved methods. The program may normalize or split the problem into fragments, create one or more subproblems, run them, inspect results, adjust modeling if necessary, and choose the final answer.

The result of each solver call must be standardized. Every `solve()` returns at least `{ status, solution, stats, diagnostics }`, where `solution` may be replaced by domain-specific fields such as `facts`, `path`, `plan`, or `assignment`. This regularity simplifies LLM prompts and lets the formatter work on a stable structure.

### 4. Leaf solver contracts

Each leaf solver resolves one local problem class and must declare three things: minimum input obligations, quickly detectable contradictions, and a complexity measure that can halt processing before blockage.

The five leaf solver classes and their contracts are:

#### 4.1 RuleProblem
For finite deduction. Minimum inputs are well-formed predicates, consistent arities, and a clear query. Quickly detectable contradictions include incompatible arities, undeclared predicates, and impossible references. Execution is limited by a maximum iteration count and a maximum number of derived facts. If the limit is reached, the result must be `too_complex` or `partial`, never a hang.

| Obligations / contradictions / limits |
| --- |
| Requires predicates and a clear query. |
| Detects incompatible arities and undeclared predicates. |
| Limits iterations and derived facts. |

#### 4.2 ConstraintProblem
For assignment and discrete puzzles. Minimum inputs are finite domains, cardinalities or uniqueness relations, and a final query. Quickly detectable contradictions include empty domains, impossible cardinalities, simultaneous `require` and `forbid` on the same instance, strict cyclic orderings, and trivial incompatibilities. Execution is limited by backtracking budget, maximum search nodes, and maximum solutions. If the raw search-space estimate exceeds a threshold, the problem is rejected before search.

| Obligations / contradictions / limits |
| --- |
| Requires finite domains and cardinalities. |
| Detects trivial contradictions and order cycles. |
| Limits search nodes and solutions. |

#### 4.3 GraphProblem
For reachability, paths, and orderings. Minimum inputs are nodes and edges (or an equivalent normalized form) plus a clear query. Quickly detectable contradictions include queries on nonexistent nodes and requests for topological order on a cyclic graph. Execution is limited by maximum nodes, maximum edges, and maximum expansions.

| Obligations / contradictions / limits |
| --- |
| Requires nodes, edges, and a clear query. |
| Detects locally impossible requests. |
| Limits graph size and expansions. |

#### 4.4 SearchProblem
For finite plans and state-space search. Minimum inputs are initial state, goal state, action vocabulary, and strategy. Quickly detectable contradictions include goals incompatible with trivial invariants and total absence of actions needed for any progress. Execution is limited by maximum depth, maximum frontier size, maximum visited states, and total time. This solver is the most exposed to combinatorial explosion, so thresholds must be strict and simple.

| Obligations / contradictions / limits |
| --- |
| Requires initial state, goal, and actions. |
| Detects trivial impossibilities. |
| Limits depth, frontier, and visited states. |

#### 4.5 NumericProblem
For bounded variables and simple arithmetic constraints. Minimum inputs are variables with finite intervals and well-formed constraints. Quickly detectable contradictions include incompatible intervals, references to undeclared variables, and evidently inconsistent inequalities. Execution is limited by the maximum product of interval cardinalities and by the maximum number of candidate evaluations.

| Obligations / contradictions / limits |
| --- |
| Requires bounded variables and constraints. |
| Detects local inconsistencies. |
| Limits evaluated combinations. |

The minimal recommended method surface for each class is:

| Class | Essential methods |
| --- | --- |
| `RuleProblem` | `fact`, `rule`, `queryFact`, `queryAll`, `solve` |
| `ConstraintProblem` | `domain`, `exactlyOne`, `atMostOne`, `require`, `forbid`, `implies`, `before`, `nextTo`, `query`, `solve` |
| `GraphProblem` | `node`, `edge`, `undirectedEdge`, `findPath`, `findShortestPath`, `findReachableFrom`, `findTopologicalOrder`, `solve` |
| `SearchProblem` | `initialState`, `goalState`, `action`, `strategy`, `maxDepth`, `query`, `solve` |
| `NumericProblem` | `variable`, `constraint`, `objective`, `query`, `solve` |

### 5. PreflightAnalyzer, ObligationRegistry, and QuestionGenerator

`PreflightAnalyzer` runs before any significant execution. It does not solve the problem; it checks whether the program is worth executing. `ObligationRegistry` is a static description of the minimum requirements for each problem class. `QuestionGenerator` transforms missing obligations into standardized questions without assuming open-ended creativity.

`PreflightAnalyzer` must walk the internal program and answer four questions:

1. Is the program valid?
2. Is the information sufficient?
3. Are there quickly detectable contradictions at the orchestration level, or must contradiction checks be delegated immediately to a leaf solver's local pre-check?
4. Does the complexity estimate fit within budget?

From this analysis it returns one of the following operational verdicts:

| Verdict | Action |
| --- | --- |
| `EXECUTE` | Start execution. |
| `NEEDS_CLARIFICATION` | Halt and produce questions. |
| `UNSAT_EARLY` | Halt and report the contradiction. |
| `INVALID_PROGRAM` | Request program regeneration. |
| `TOO_COMPLEX` | Halt and report limit exceeded. |

`ObligationRegistry` must be explicit and static. For each leaf solver, the registry declares which fields are mandatory, which relations are required for modeling, and which minimum and maximum thresholds exist. This enables symbolic checks before search and makes the system predictable.

`QuestionGenerator` must not invent questions freely. It uses templates tied to obligations. If a relation cardinality is missing, the question asks for the cardinality. If an order type is missing, the question asks whether the order is linear or circular. If a numeric bound is missing, the question asks for the relevant finite interval. Thus questions are structurally determined and can be generated even without an LLM. The LLM may be used only for more natural phrasing.

### 6. Fail-fast and blocking-control mechanisms

To avoid infinite loops or long stalls, the design assumes that any solver may fail honestly and early. No solver is allowed to run "until it finds something". All executions must have a time budget and a step budget. In addition, each solver has its own structural thresholds.

There are two control levels:

1. **Static** — `PreflightAnalyzer` halts upfront cases that are obviously invalid, contradictory, or too large.
2. **Dynamic** — each solver increments a step counter and periodically checks its deadline. On exceedance, the solver halts and returns `too_complex` or `partial`; it must not wait.

Recommended control measures:

| Mechanism | Role |
| --- | --- |
| Maximum time per request | Halts global execution. |
| Maximum time per solver | Prevents monopolization by a single subproblem. |
| Maximum step count | Controls cost of iterative algorithms and search. |
| Maximum frontier size | Prevents explosion in `SearchProblem`. |
| Maximum derived facts | Prevents uncontrolled closure in `RuleProblem`. |
| Maximum backtracking nodes | Limits `ConstraintProblem`. |
| Maximum product of domains | Halts bounded numeric and some discrete problems before execution. |
| Maximum solution count | Avoids useless enumeration when the final answer does not require it. |

It is acceptable, and encouraged, for the system to halt a problem with status `too_complex`. The final message must state directly that the problem exceeds configured limits and that the result was not fully computed. This behavior is preferable to an execution that runs too long or to an unverified conclusion.

### 7. Complete processing flow

The processing flow is concrete and separated by role:

1. **Formatting instructions inference** — deduce output style from the user request.
2. **Program generation** — ask the LLM for either a `SolverProgram`, clarification questions, or an explicit refusal to formalize.
3. **Preflight analysis** — validate the program, detect missing obligations, detect quick contradictions, and estimate complexity.
4. **Verdict routing**:
   - `INVALID_PROGRAM` → request a single local regeneration using the diagnostic.
   - `NEEDS_CLARIFICATION` → halt and return the generated questions.
   - `UNSAT_EARLY` → report the contradiction.
   - `TOO_COMPLEX` → report that limits are exceeded.
   - `EXECUTE` → run the program.
5. **Symbolic execution** — run the program inside `ExecutionContext`, calling leaf solvers as needed.
6. **Result formatting** — send the structured result to the formatter LLM, which produces the final text without recalculating the solution.

This scheme separates roles cleanly. The LLM translates and phrases. Symbolic analysis validates, detects missing information, and fails early. Local solvers solve only what is already sufficiently specified. The formatter explains only what resulted.

### 8. LLM prompt boundaries

Prompts must be few and strictly separated.

| Purpose | Prompt requirement |
| --- | --- |
| Program generation | Return `PROGRAM`, `QUESTIONS`, or `FAIL`; use only the approved DSL; set the final value explicitly via `ctx.setFinal(...)`. |
| Regeneration after errors | Repair the program minimally based on the diagnostic; do not rewrite unnecessarily. |
| Style inference | Return a compact JSON object with fields such as `style`, `brevity`, `include_steps`, `include_assumptions`, `include_alternatives`; do not solve the problem. |
| Final result formatting | Treat the execution result as the source of truth; do not recompute; if the result is `failed` or `partial`, explain that directly and briefly. |
| Formalization fallback (optional) | If the program cannot be built after allowed retries, produce a short user-facing response explaining that the symbolic runtime could not formalize the request precisely enough, indicating whether the difficulty was ambiguity, unsupported structure, or search complexity; do not fabricate a solution. |

The generation prompt must explicitly allow the model to refuse to produce a program. This solves the problem symbolically and procedurally: the LLM must not be forced to emit a bad program when the problem is clearly incomplete or too ambiguous. At the same time, the runtime must not blindly trust that judgment; it runs its own filter whenever a program is proposed.

### 9. Composing subproblems through context

Composition between different subproblem types is not a declarative operation sufficient by itself. Composition is treated as an orchestration program, but with a small set of constructions and checks so that the system does not get stuck in arbitrary executions or endless searches.

The `ExecutionContext` is the explicit linkage mechanism. A program may:

1. Run a `RuleProblem` to derive transitive relations or closed facts.
2. Store the result with `ctx.storeResult(name, result)`.
3. Project needed fields from that result into context variables with `ctx.project`, `ctx.map`, or `ctx.filter`.
4. Pass those variables as inputs to a `ConstraintProblem`, `GraphProblem`, `SearchProblem`, or `NumericProblem`.
5. Inspect the second result, assert preconditions, and either chain further solvers or call `ctx.setFinal`.

This explicit step-by-step composition is the analog of a human solver writing down intermediate results and reusing them in the next stage. The DSL does not hide this chaining behind a single declarative keyword; it makes the data flow visible in the program. Because the context is typed and named, the preflight analyzer can trace which solver consumes which variables and reject programs with dangling references or type mismatches before execution starts.

## Decisions & Questions

Question #1: Why is the public API kept to a single function `solveText(llmAdapter, text) -> text` while the internal architecture is significantly richer?

Response: Simplicity at the boundary preserves usability. The internal complexity is justified by the need to handle natural-language requests that contain nuance, implicit assumptions, and mixed reasoning types. Hiding that complexity behind one function lets callers treat the solver as a black box that translates text to text, while the runtime maintains full auditability internally.

Question #2: Why does the DSL remain syntactically JavaScript but semantically a narrow orchestration language?

Response: Using JavaScript syntax reduces friction for the LLM generating the program and for runtime parsing. Restricting semantics to a bounded orchestration language prevents the LLM from introducing uncontrolled loops, arbitrary algorithms, or hidden side effects. The runtime can therefore validate and execute the program predictably.

Question #3: Why is `ExecutionContext` mandatory and strongly typed rather than a plain dictionary?

Response: A plain dictionary would make it impossible to distinguish user-given facts from derived values or temporary assumptions. Typed registers (`given`, `vars`, `results`, `assumptions`, `unknown`, `conflicts`, `trace`, `final`) make the program state inspectable, enable preflight checks for dangling references, and let the formatter produce honest explanations of what was assumed versus what was proven.

Question #4: Why does each leaf solver declare its own obligations, quick contradictions, and complexity limits?

Response: Centralizing all checks in one place would create a bottleneck and would force the preflight analyzer to know internals that belong to the solver. By making each solver declare its contract, the system stays modular: new solver classes can be added without rewriting the analyzer, and the analyzer can query each solver's obligation registry generically.

Question #5: Why are there two levels of fail-fast control (static preflight and dynamic step/deadline counters)?

Response: Static preflight catches structural impossibilities before any costly work begins. Dynamic counters catch runtime explosion that static analysis could not predict, such as unexpectedly large intermediate collections or edge-triggered search blow-up. The combination ensures that neither obvious mistakes nor pathological inputs can stall the system.

Question #6: Why is `TOO_COMPLEX` an encouraged, first-class status rather than a failure to avoid?

Response: Pretending to solve a problem that exceeds budget is worse than admitting the limit honestly. Returning `too_complex` preserves trust, keeps response times bounded, and gives the caller a clear signal that either the problem needs reformulation or the runtime limits must be raised.

Question #7: Why must the formatter LLM treat the execution result as source of truth and never recompute?

Response: Separating computation from presentation prevents the LLM from hallucinating a different answer than the one the symbolic runtime verified. The formatter's only job is to translate the structured `final` object into natural language according to the inferred style. This preserves correctness and traceability.

Question #8: Why is composition between subproblems done through explicit context steps rather than a single declarative composition primitive?

Response: Explicit steps make the data flow auditable. A human reading the program can see exactly which intermediate result feeds which solver. It also lets the preflight analyzer verify reference validity and type compatibility between stages. A hidden declarative composition would obscure these checks and make debugging harder.

## Conclusion

`logic-eval` is the bounded symbolic solver layer of MRP-VM v0. It replaces open-ended LLM reasoning with a controlled pipeline: natural language is translated into a narrow internal DSL, the DSL program is validated by a preflight analyzer, executed inside an explicit typed context, and resolved by small local solvers with clear contracts and strict budgets. The result is then formatted by an LLM that treats the runtime output as source of truth.

The design does not hide its limits. Problems that are too ambiguous, too large, or too complex for the configured budgets are halted early and reported honestly. Where the request is sufficiently structured for local formalization but nuanced enough to benefit from LLM translation and orchestration, the architecture provides decomposition, working memory, strategy selection, local verification, and final reformulation.

The correct evolution path is not to expand `logic-eval` into a universal solver, but to keep the public API minimal, keep the internal DSL strict, and enrich the system gradually by adding new leaf solver classes, new context operations, and new expression operators without changing the outer contract.
